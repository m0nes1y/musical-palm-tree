## 四、登录流程（重点）

登录不是简单表单提交，而是一套**多步 + 行为验证码**流程（前端 JS 完整可见）：

```
1. 用户输入 用户名 + 密码
2. GET  time.php          → 返回 { token, timestamp }
3. POST Verificationcode.php (token, timestamp)
                            → 返回 验证码图片(base64 PNG) + 点击顺序提示 + captcha_id
4. 用户在 Canvas 上「按指定顺序点击 4 次」(行为/认知验证码)
5. POST 当前页 "" (clicks:[{x,y}×4], captcha_id, username, password)
6. 返回 code===0 → 跳转 /index.php（登录成功）
```

**验证码类型**：自定义 Canvas 点击序列验证码（"请按顺序点击" + 4 次坐标点击），非传统字符验证码。

## 五、客户端安全/隐私相关实现（技术事实）

1. **浏览器指纹**：`getBrowserFingerprint()` 采集
   - 屏幕分辨率/色深、时区、语言、平台
   - Canvas 指纹、WebGL 指纹（RENDERER/VENDOR）、插件列表
   - 整体 base64 编码

2. **弱"加密"**：`simpleEncrypt()` 是 **XOR 异或 + base64**——非标准加密，可轻易还原，仅为混淆。

3. **凭据传输**：用户名/密码以 **明文 JSON** 经 POST 发送（页面为 HTTPS，传输层有 TLS 保护，但应用层未做额外加密）。

4. **无防暴力破解可见措施**：登录接口、验证码接口均为同域 PHP 文件，未见限流/锁定的前端痕迹。

> 说明：以上为前端代码可见事实，不代表服务端实现。是否真实安全需服务端审计。

## 七、分析结论

- **性质**：一个登录门禁式平台，公开面仅登录页；业务功能（"防营销骚扰"具体指什么）完全在鉴权之后，无法从公开页面确认。
- **技术成熟度**：中等。采用成熟的 layuiadmin 后台框架 + 自定义行为验证码 + 客户端指纹，说明有一定前端工程投入。
- **可抓取性**：公开页极易抓取（15KB 静态结构）；但**核心数据必须登录**，且登录带行为验证码 + 指纹，自动化登录门槛高。
- **风险提示（技术视角）**：客户端 XOR "加密"形同虚设，若服务端同样未做加固，账号体系存在潜在薄弱环节；建议仅作技术评估，勿用于未授权访问。
